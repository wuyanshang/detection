# 语义歧义检测 · 门控重构方案

> 目标读者：本检测链路的开发与规则维护者
> 适用范围：数据字典字段元数据（表中英文名 + 字段中英文名 + 缩写映射）的语义歧义检测
> 文档定位：在现有两阶段（stage1 消歧 + stage2 分类）基础上的**保守版重构**，附代码门控层规格与 prompt 改动清单

---

## 1. 背景：为什么要改

### 1.1 演进史与已踩的坑

| 版本 | 形态 | 结果 / 问题 |
|------|------|------|
| v1 | 只写 B000001~B000004 几条规则，单趟跑 | 违反率（误报）压不下去，万物皆可挑刺 |
| v2 | 消歧 + 规则**写在同一个 prompt** (`history/semantic-ambiguity.txt`) | 消歧能压违反率，但 AI **偷看后面的规则**，把消歧带偏（为了能报，故意判消歧失败） |
| v3 | 物理拆成两次调用：`stage1-fixed` + `stage2-fixed` | 偷看解决；但仍有两个新问题，见下 |

### 1.2 v3 仍存在的两个问题

1. **AI 发散 / 误报**：
   - `代码类型` 等字段在 stage2 触发 B000002，原因是 LLM 把 `代码`/`类型` 脑补成高风险词表里的近义词（prompt 里"精确匹配、不得推断近义词"是软约束，LLM 不可靠地遵守）。
   - 大量字段触发 B000003（中文名无法理解），因为该规则判定标准开放式（"指代对象不够精确"），没有任何前置门控。

2. **stage1 的 D1 自相矛盾 → 多次运行结果不一致**：
   - D1 要求 `field_cn 是可理解的中文业务描述，指代对象明确`，这一句把**两个标准不同的问题**焊在一起：
     - 问题 A：**能不能理解**（这字段大概在说什么）—— 消歧该问的
     - 问题 B：**够不够具体 / 有没有业务限定**（指代精不精确）—— B000002 该问的
   - `金额信息` 这类字段对 A 答"能理解"、对 B 答"不够具体"，D1 强行二合一，AI 只能输出"字段可理解但业务含义模糊"的自相矛盾，反复横跳。
   - 且 D1 的"步骤1 高风险词表前置检查"与 stage2 的 B000002 **完全重复**，同一判断做两次，是不一致的第二来源。

---

## 2. 设计原则

1. **能理解 ≠ 够规范，二者正交。** 字段可以"能理解但命名不规范"（如 `金额信息`）。不能再用"消歧成功就豁免一切规则"把这两件事绑死。
2. **门控（要不要查）与判定（违不违规）分离。** 门控尽量用确定性代码做，判定才交给 LLM。
3. **只做减法，保证单调性。** 相对 v3，本方案只在前面加确定性门控、缩小可报范围，违反率只会 ≤ v3，结构上不可能退回 v1。
4. **保留 stage1 物理独立。** 不与规则同 prompt，杜绝偷看；保留其 D2/D3/D4 的表名整体推断能力。
5. **词表唯一、精确匹配。** 高风险词表全链路只有一份，纯字面精确匹配，代码执行，不让 LLM 判断是否命中。

---

## 3. 核心认知：四条规则按"问题性质"分两类

| 规则 | 本质问题 | 归属 |
|------|----------|------|
| **B000001** 中英文矛盾 | 能不能理解 | **理解类** → 由 stage1 消歧把关 |
| **B000003** 枚举值无字段含义 | 能不能理解 | **理解类** → 由 stage1 消歧把关 |
| **B000002** 缺业务限定 | 能理解、但命名不够具体 | **规范类** → 由代码门控独立路由 |
| **B000004** 中文名缩写多义 | 能理解、但命名不够清晰 | **规范类** → 由代码门控独立路由 |

- **理解类**：`消歧成功 = 能理解` → 自动不报；只有 `消歧失败` 才进 stage2 判这两条。
- **规范类**：与"能否理解"无关，**命中代码门控即进 stage2 判定，不被"消歧成功"豁免**。这是相对 v3 的关键解耦动作。

---

## 4. 整体架构

```
每个字段
  │
  ▼
① A 规则（确定性，已有）：空 / 纯英文 / 乱码 / 重复
  │   命中 → 拦截，不进语义检测
  ▼ 通过
② 代码门控层（新增 NodeAction，纯代码，无 LLM）
  │   · 命中备用词表 → 标记 B000005 备用类字段待核实，短路（不进 stage1/stage2）
  │   计算"规范类候选"：
  │     · hitB2 = field_cn 精确命中高风险词表
  │     · hitB4 = field_cn 含英文字母数量 ≥ 2
  ▼
③ stage1 消歧（LLM，保留，去掉高风险词检查）
  │   仅决定理解类(B000001/B000003)：
  │     · 消歧成功 → 理解类不报
  │     · 消歧失败 → 理解类进入 stage2
  ▼
④ 汇总 stage2 判定集合 =
  │     {消歧失败时的 B000001/B000003}
  │   ∪ {hitB2 → B000002}
  │   ∪ {hitB4 → B000004}
  │
  ├─ 集合为空 ─▶ has_ambiguity=false，不调 stage2
  │
  └─ 集合非空 ─▶ ⑤ stage2 分类（LLM，按需加载）
                  只把集合内规则的判定标尺喂给 LLM，
                  LLM 看不到集合外的规则，无从偷看、无从发散
```

**为什么不会复发老问题**

| 老问题 | 会不会有 | 原因 |
|--------|----------|------|
| v1 误报洪水 | 不会 | 单调性：消歧照旧 + 门控只减不加，可报集合 ⊆ v3 |
| v2 AI 偷看 | 不会 | 保留物理拆分；stage2 按需加载，集合外规则根本不在 prompt 里 |
| 新风险：漏报 | 可能，但可控 | 见 §10 |

## 5. 代码门控层规格（新增节点）

### 5.1 节点定位

在 workflow 中新增一个 `GateNode`（`NodeAction`），位于 A 规则节点之后、stage1 调用之前。它给每个 `FieldResult` 计算并暂存"规范类候选"标记，供后续 stage2 路由使用。

### 5.2 唯一高风险词表（封闭列表，精确字面匹配）

```
姓名 / 状态 / 日期 / 金额 / 关系 / 编号 / 利率 / 地址
```

> 全链路只此一份；三个旧文件的不一致（stage1 含"地址"、stage2/历史版不含）以本表为准，统一为含"地址"的 8 个词。
> 匹配规则：`field_cn` 作为字符串，**子串包含**词表中任一词即命中。仅做字面比较，**不得推断近义词 / 同义词**（这正是 `代码类型` 不再误命中的根因——它字面不含上述任何词）。

### 5.3 各规范类规则的机械门控条件

| 候选标记 | 命中条件（纯代码可判定） | 说明 |
|----------|--------------------------|------|
| `hitB2`（B000002 候选） | `field_cn` 子串命中高风险词表 | 命中才允许 stage2 判 B000002 |
| `hitB4`（B000004 候选） | `field_cn` 中英文字母**数量 ≥ 2** | 缩写至少 2 个字母（如 TP、WP）；单个字母多为序号（如 `A类`、`1区`），不算缩写，故 ≥2 才进。纯英文已被 A 规则拦掉 |

> 理解类（B000001/B000003）**不需要**代码词表门控：它们由 `消歧失败` 这道强过滤天然收口（绝大多数字段消歧成功），再叠加各自的窄定义（见 §7）。
> B000001 走到判定时必然 `field_en 非空且可读`（因为消歧只在 field_en 可读时才会因"严重矛盾"失败），无需额外门控。

### 5.4 伪代码（贴合现有 NodeAction 风格）

```java
@Slf4j
@Component
public class GateNode implements NodeAction {

    // 唯一高风险词表
    private static final List<String> HIGH_RISK_WORDS =
        List.of("姓名", "状态", "日期", "金额", "关系", "编号", "利率", "地址");

    @Override
    @SuppressWarnings("unchecked")
    public Map<String, Object> apply(OverAllState state) {
        List<FieldResult> results = (List<FieldResult>) state.value("results").orElseThrow();

        for (FieldResult r : results) {
            // 已被 A 规则拦截的字段不再计算门控
            if ("拦截".equals(r.getQualityResult())) continue;

            String fieldCn = r.getFieldCn() == null ? "" : r.getFieldCn();

            boolean hitB2 = HIGH_RISK_WORDS.stream().anyMatch(fieldCn::contains);
            // B000004：中文名含英文字母数量 ≥ 2 才进（单字母多为序号，不算缩写）
            long letterCount = fieldCn.chars()
                    .filter(c -> (c >= 'A' && c <= 'Z') || (c >= 'a' && c <= 'z'))
                    .count();
            boolean hitB4 = letterCount >= 2;

            r.setGateB2(hitB2);   // 需在 FieldResult 上新增字段
            r.setGateB4(hitB4);
        }
        return Map.of("results", results);
    }
}
```

> `FieldResult` 需新增两个布尔位（如 `gateB2` / `gateB4`），或用一个 `Set<String> candidateRules` 承载允许清单，按团队习惯择一。

### 5.5 备用类字段待核实（B000005，新增确定性规则）

**目的**：识别备用 / 预留 / 占位类字段并标记 `B000005 备用类字段待核实`，提示人工核实（这类字段往往是占位、待清理或命名不规范）。

**为什么放在确定性层**：备用类字段由关键词即可判定，**纯代码、不调 LLM**。命中即**短路**——直接标记 B000005，**不再进 stage1/stage2**（占位字段无需做语义消歧）。

**判定（命中任一即触发）**：

| 来源 | 匹配方式 | 词表（medium 范围） |
|------|----------|---------------------|
| `field_cn` | **子串**包含 | 备用 / 预留 / 占位 / 保留 / 扩展 / 冷备 / 备份 / 预备 / 待用 / 冗余 |
| `field_en` | 按驼峰/下划线/数字切词后**整词**匹配（不可子串） | spare / reserve / reserved / rsv / rsrv / filler / fill / bak / backup / ext / pad / padding / unused / dummy |

> **英文必须整词匹配**：否则 `ext` 会误伤 `context`/`next`/`external`，`pad` 会误伤 `update`。切词后整词比较可避免。
> **中文 `保留`/`扩展` 有误伤风险**（如 `保留比例`、`扩展信息`）。但本规则定位是"**待核实**"——输出给人工复核而非自动拦截，少量误报可接受；若要更严，可把 `保留`/`扩展` 移出或要求其后接"字段/位/区"等占位形态词。

**伪代码（并入 GateNode，置于 hitB2/hitB4 计算之前）**：

```java
private static final List<String> SPARE_CN =
    List.of("备用", "预留", "占位", "保留", "扩展", "冷备", "备份", "预备", "待用", "冗余");
private static final Set<String> SPARE_EN = Set.of(
    "spare","reserve","reserved","rsv","rsrv","filler","fill",
    "bak","backup","ext","pad","padding","unused","dummy");

// ... 在 for 循环内、A 规则放行之后：
String fieldEn = r.getFieldEn() == null ? "" : r.getFieldEn().toLowerCase();
boolean spareCn = SPARE_CN.stream().anyMatch(fieldCn::contains);
boolean spareEn = java.util.Arrays.stream(fieldEn.split("[^a-z]+"))  // 驼峰已转小写，按非字母切词
        .anyMatch(SPARE_EN::contains);
if (spareCn || spareEn) {
    r.addQualityRule("B000005", "备用类字段待核实");
    r.setSemanticDone(true);   // 短路标记：跳过 stage1/stage2
    continue;
}
```

> 注：英文切词若需处理驼峰（`bakField`），先在 `split` 前把大写转成分隔点，或用 `replaceAll("([a-z])([A-Z])","$1 $2").toLowerCase()` 再切。

## 6. stage1（消歧）改动清单

文件：`data-check/stage1-fixed.txt`

### 6.1 删除：D1 的"步骤1 高风险词表前置检查"整段

删掉现 `### D1` 下"步骤1：高风险词表前置检查（精确匹配，禁止推断）"的全部内容（含示例）。**理由**：该段做的是 B000002 的活（缺业务限定判断），与 stage2 重复，且把"够不够具体"混进了"能不能理解"，是 D1 自相矛盾与多次运行不一致的根源。

### 6.2 删除：D3 的"高风险词追加校验"

删掉现 `### D3` 下"步骤4：高风险词追加校验"（即依赖 D1 高风险词失败标签、再用表名补限定的那段）。删掉步骤1 后它失去依赖，且同属 B000002 范畴。

### 6.3 改写：D1 适用条件的措辞

把 D1 判定从"指代对象明确"下调为"能判断字段大致业务含义"，让 stage1 只回答"能不能理解"：

```text
### D1：字段中文名可理解且与英文名无矛盾

**适用条件**：field_cn 是可理解的中文业务描述，结合上下文能判断字段**大致业务含义**（不要求精确到业务限定，"够不够具体"由 stage2 的 B000002 负责，本阶段不评判）。

**判断步骤**：检查 field_en 是否与 field_cn 严重矛盾
- field_en 为空，或为缩写组合（如 CLMLSID, INSNAM）→ 无法判断矛盾，D1 通过
- field_en 为可读英文 → 检查是否严重矛盾：
  - 轻微不一致（同一业务实体）→ D1 通过（例：field_cn=`保单号`, field_en=`policyNumber`）
  - 严重矛盾（指向完全不同的业务实体）→ D1 失败（标签：`D1失败-严重矛盾`）
```

### 6.4 明确：stage1 的产出只服务于理解类

在 stage1 的执行规则里补一句定位说明（不改变输出格式）：

```text
> 本阶段判定的"消歧成功/失败"仅用于决定理解类规则（B000001 中英文矛盾、B000003 枚举值无含义）是否需要进入 stage2。
> "缺业务限定"(B000002)、"缩写多义"(B000004) 属命名规范问题，由代码门控独立路由，不在本阶段评判，也不因本阶段"消歧成功"而被豁免。
```

### 6.5 D2 / D4 保持不变

D2（中英文互补）、D4（技术类字段）原样保留，表名整体推断能力（D3 的核心，去掉高风险词那段后剩下的部分）保留。

## 7. stage2（分类）改动清单

文件：`data-check/stage2-fixed.txt`

### 7.1 入参新增"待判规则集合"，按需加载

stage2 不再固定检查 4 条规则，而是只检查调用方（由 §4 ④ 汇总）传入的集合。建议在 `<input>` 增加一行：

```text
待判规则: {rules_to_check}   # 例如 ["B000002"] 或 ["B000001","B000003"]
```

并在任务说明中强调：

```text
只判断"待判规则"列表中的规则是否触发，**不得检查或提及列表之外的任何规则**。
```

**已落地为真·按需加载（而非软约束）**：
- `prompts/ambiguity-stage2.txt` 是**基底模板**，含占位符 `{rules_block}`；
- 每条规则的判定标尺拆成独立片段 `prompts/rules/B000001.txt … B000004.txt`（可单独维护/配置）；
- 代码（`AmbiguityLlmClient#buildRulesBlock`）按"待判规则"只把命中规则的片段拼进 `{rules_block}`，**未命中规则的文字根本不出现在 prompt 里** → LLM 无从看到、无从偷看越界。
- 双保险：`Stage2Node` 对返回再做代码级过滤，只接受 `ruleCode ∈ 待判规则` 的命中。

### 7.2 B000002：保留判定标尺，删除前置过滤段

- **删除** B000002 现有的"前置过滤（高风险词表）"段——该门控已上移到代码层（§5）。
- **保留** 排除条件（尤其 `table_cn 已限定唯一业务场景 → 不报`，这是表名推断在本规则内的体现）。
- 取消其"必须先于一切、依赖消歧失败"的隐含前提：B000002 现在由 `hitB2` 直接路由，**与消歧成功与否无关**。
- **用下列功能性定义替换原来那段封闭的"业务限定修饰语"列表**（列表永远列不全，改用原则判断）：

```text
**什么是业务限定（功能性定义，优先于任何词表）**

业务限定 = 能回答"这个词是『谁的 / 哪个业务对象的 / 什么业务环节的 / 哪种业务量』"
的修饰成分，它把宽泛的高风险词收窄到唯一的业务指代。

功能性判定法：把修饰词去掉，指代范围是否变宽？
- 变宽（修饰词承载了"区分是哪一个业务对象"的信息）→ 它是业务限定
- 不变（修饰词没收窄任何业务指代）→ 不是业务限定

业务限定的四类（开放示例，非穷举）：
- 业务主体（谁的）：投保人、被保人、受益人、客户、保单持有人、代理人/业务员、机构/网点、再保人、经办/复核人 …
- 业务场景/环节（什么环节）：投保、承保、核保、理赔、给付、退保、保全、续期、回访、对账 …
- 业务属性/科目（哪种量）：保费、赔款、佣金、现金价值、准备金、红利、利息、手续费、税 …
- 险种/产品维度（哪类产品）：险种、产品、主险、附加险 …

不算业务限定：技术词（字段/信息/数据/记录/表）、序号（1/2/A）、助词（的/相关/对应）、
以及泛化词（其他/业务/相关）和本身不指向具体业务对象的结构词（代码/类型/标志，
除非其前已接实体，如"保单状态"中的"保单"）。

**重要：不是"前面有个词"就算限定**，必须该词属上述四类且确实收窄了指代。
- 仍报（有词但未收窄）：`金额字段1`、`金额信息`、`其他金额`、`相关金额`、`业务金额`
- 不报（有业务归属）：`投保人金额`、`理赔金额`、`保费金额`

**不要过严**：有了业务归属即放行，不要求精确到唯一科目。
如 `投保人金额` 虽未区分应缴/已缴，但已有"投保人"主体归属 → 不报。
```

### 7.3 B000004：保留判定标尺，删除前置过滤段

同理，"仅适用于 field_cn 含英文缩写"的门控上移到代码层（`hitB4`）。保留"即使结合所有信息仍多种展开"的判定标准与白名单排除。由 `hitB4` 直接路由。

### 7.4 B000003：收窄为"枚举值无字段含义"

重定义并替换现有 B000003：

```text
## B000003 - 字段中文名是枚举取值、不承载字段含义

**前提**：本规则仅在 stage1 消歧失败时，且被列入"待判规则"时才检查。

**判定标准**：
field_cn 看起来是某个**取值 / 枚举值本身**，而不是对"字段存什么"的描述。
即：它回答了"值是什么"，却没回答"这是什么字段"。

**报**：
- field_cn=`男` / `已婚` / `借方` / `正常` —— 是取值，不是字段名
**不报**：
- field_cn=`性别` / `婚姻状态` / `借贷方向` —— 是字段名（描述了维度）
- field_cn 含分类维度词且维度清晰（如 `险种类型`）—— 能理解，消歧阶段已通过
- field_cn=`代码类型` —— 有"代码"作为区分维度，视为能理解，不报
```

> 说明：B000003 不设代码词表门控，其数量由"消歧失败"(强过滤) + 上述窄定义共同压制，天然不会洪水。

### 7.5 B000001：保留，明确前提

保留现判定（中英文指向不同业务概念 + field_en 可读前置判断）。明确其前提为"stage1 消歧失败（含 `D1失败-严重矛盾`）且被列入待判规则"。

### 7.6 删除 override_logic

stage2 现有的 `<override_logic>`（"全部未触发则覆盖 stage1 消歧失败判定"）可**删除**。新架构下：
- 理解类只在消歧失败时才进 stage2，是否成立由具体规则判定，不需要反向覆盖 stage1；
- 规范类与消歧无关，不存在覆盖问题。

## 8. 多次运行一致性（担心点 1）

**问题**：同一字段（相同 table_cn/table_en/field_cn/field_en）多次运行，结果是否相同，包括触发的规则？

**诚实结论**：LLM 调用本质上非确定，**不做处理则不保证一致**；但本架构 + 下列三招后，可达到"相同输入实质 100% 一致"。

### 8.1 架构本身已消除最大的不一致来源

"触发哪些规则"分两步：**门控/路由（要不要查）** 和 **判定（查出来违不违规）**。

- **门控/路由是纯代码**（§5：`hitB2`/`hitB4`、消歧失败后的理解类入选），**每次完全一致**。这一步消除了 v3 时代最大的变量——原 stage1 用模糊的"高风险词 + 业务限定"判断来路由，本身就来回翻，导致候选规则集合每次都变。现在候选集合是确定性的。
- 剩余的不一致只可能来自 **LLM 在已确定范围内的判定**。

### 8.2 三招把 LLM 判定也压到高度一致

| 手段 | 做法 | 效果 |
|------|------|------|
| **1. temperature=0（首要）** | 当前 `LlmSensitiveDetector` 调用未设 temperature，用的是 qwen-plus 默认值（偏高，约 0.7），这是不一致的**头号原因**。改为贪心解码 | 同输入输出高度稳定 |
| **2. 判定标尺清单化** | 规则判定尽量写成"是否含 X / 是否属某类"的二元清单（B000003 窄定义、B000002 明确排除已在做），缩小边界摇摆带 | 边界样本翻转概率大降 |
| **3. 结果缓存（彻底解决）** | 以 `md5(promptVersion + table_cn + table_en + field_cn + field_en)` 为 key（**不含缩写**）；**白名单：只缓存"无歧义(通过)"的字段**，命中即跳过 LLM | **相同输入 100% 一致**，且省调用 |

> 白名单语义：命中 = 已知良好，直接放行；未命中 = 新字段或曾被标记(有歧义)，需重判。有歧义项**不入缓存**——它们待修复，缓存会变过期。这也让缓存表只增不更、体积受"去重后的良好字段数"约束。

### 8.3 temperature=0 的具体改法

`LlmSensitiveDetector#detect` 的调用处补上 options（Spring AI / qwen 兼容写法）：

```java
import org.springframework.ai.openai.OpenAiChatOptions;

String response = ChatClient.create(chatModel)
        .prompt()
        .user(prompt)
        .options(OpenAiChatOptions.builder()
                .temperature(0.0)
                .seed(42)        // qwen 兼容模式支持；进一步增强可复现性
                .build())
        .call()
        .content();
```

> 做了第 1 招后，未命中缓存时也高度一致（非数学保证）；叠加第 3 招后，**命中缓存即数学保证一致**。建议两者都上。

## 9. 单条数据的规则组合是否合理（担心点 2）

**问题**：一条数据触发多条规则时，组合是否合理、会不会自相矛盾或冗余？

### 9.1 可能的组合与判断

解耦后，一个字段最终可能落入的规则来自两类：

- 理解类（消歧失败才有）：B000001 中英文矛盾、B000003 枚举值无含义
- 规范类（门控命中即有）：B000002 缺业务限定、B000004 缩写多义

| 组合 | 是否合理 | 说明 |
|------|----------|------|
| 单条规则 | ✅ 合理 | 最常见 |
| B000002 + B000004 | ✅ 合理 | 如 `TP金额`：既缺业务限定又含多义缩写，确是两个真问题 |
| B000001 + B000003 | ⚠️ 冗余 | 都在说"中文名层面有问题"，同时报会显噪 |
| **理解类 + 规范类同时报** | ❌ 不合理 | 字段连"是什么"都没搞清（理解类触发），却又去报它"命名不够规范"（规范类），口径打架 |

### 9.2 治理：理解类优先，确定性后处理抑制规范类

在 stage2 结果之上，加一段**纯代码后处理**（确定性，不影响 §8 一致性）：

```
若本字段触发了任一理解类规则（B000001 或 B000003）：
    → 抑制规范类（B000002 / B000004）的输出
    → 即：一条数据要么报"读不懂类"，要么报"命名规范类"，不混报
```

**理由**：能不能理解 → 够不够规范，是递进关系。先把"读不懂/矛盾"解决了，再谈命名规范才有意义。这样每条数据的结论始终聚焦在最根因的那一层，组合天然合理。

> 该抑制只在"消歧失败 + 理解类成立"时触发；消歧成功的字段本就只可能报规范类，不受影响。

**理解类内部只报一条**：B000001 与 B000003 都在指"中文名层面有问题"，属冗余，二者只报一条，优先级 **B000003 > B000001**（"中文名根本不是字段名、只是个取值"比"中英文矛盾"更根本）。
**规范类内部可共报**：B000002（缺限定）与 B000004（缩写多义）是两个独立命名问题，可同时报，不冗余。

最终：一条数据 **要么报理解类的 1 条，要么报规范类的 1~2 条**，二者不并存。

## 10. 风险与缓解

本方案的失败模式从"误报"转为"**漏报**"（更安全、更可调）：

| 风险 | 成因 | 缓解 |
|------|------|------|
| B000002 漏报 | 高风险词表不全，该报的字段词不在表里 | 拿真实数据灰度，对比新旧结果差集，按需补词表 |
| B000003 漏报 | 窄定义后，"非枚举值但确实读不懂"的字段不再报 | 符合预期取舍（优先消灭误报）；如需可后续单列规则 |
| 规范类与消歧判断口径不一致 | 同一字段消歧成功但 B000002 报 | 属正常解耦结果（能理解≠够规范），非缺陷 |

**上线前必做**：用一批真实字段（尤其历史误报样本）同时跑 v3 与新方案，核对：
1. 新方案可报集合是否 ⊆ v3（应满足，否则有 bug）；
2. 被新方案"少报"的字段是否确为可接受的漏报；
3. `代码类型` 等历史误报样本是否已不再触发。

## 12. 新增规则指南（怎么加一条新规则）

加新规则前，先回答一个问题：**这条规则在问"能不能理解"，还是"能理解但命名不规范"？** 答案决定它属哪一类，影响完全不同。

### 12.1 判断归属

| 这条规则的本质 | 归属 | 是否要动 stage1 |
|----------------|------|-----------------|
| 关于"字段读不读得懂 / 中英文矛不矛盾 / 是不是取值"（像 B000001、B000003） | **理解类** | **要**，在 stage1 封闭清单加一条 Fn |
| 关于"能理解、只是命名不够具体/清晰/规范"（像 B000002、B000004） | **规范类** | **不动 stage1** |
| 纯关键词即可判定、根本不需要 LLM（像 B000005 备用字段） | **确定性类** | 不动 stage1/stage2，只加门控短路 |

### 12.2 配方 A：加「规范类」规则（不碰 stage1）

最常见、最简单。以新增一条 `B0000xx` 规范类规则为例：

1. **门控**：在 `AmbiguityGate` 加一个命中条件（纯代码可判定，如命中某词表/某字符特征），算出 `hitBxx`；在 `AmbiguityResult` 加对应标记位（或塞进 `candidateRules`）。
2. **片段**：新建 `prompts/rules/B0000xx.txt`，写它的判定标尺（务必带"拿不准默认不报"兜底，保持单调）。
3. **登记顺序**：把 `B0000xx` 加进 `AmbiguityConstants.STAGE2_RULES`（决定拼接/处理顺序）。
4. **路由**：在 `Stage2Node.computeRulesToCheck` 里，`hitBxx` 命中时把它加入待判集合（归到**规范类**集合）。
5.（可选）若它与其他规范类互斥/冗余，在 §9 的后处理里加抑制规则。

> stage1 一行不用动——因为规范类与"能否理解"无关，不由消歧把关。B000005 就是这条路线的极简版（第 1 步命中即短路，连片段都不用）。

### 12.3 配方 B：加「理解类」规则（要在 stage1 补一条 Fn）

理解类规则由 stage1 消歧统一把关，所以**必须同步 stage1 的封闭失败清单**，否则会出现"stage1 判消歧成功、新规则却永远进不去"或反过来"消歧失败但没有对应规则可判"的不对齐：

1. **stage1**：在 `ambiguity-stage1.txt` 的 `<decision>` 封闭清单里加一条失败条件 **Fn**，描述"这种读不懂长什么样"（要写成可判定的封闭描述，不要开放式）。
2. **片段**：新建 `prompts/rules/B0000xx.txt`，写判定标尺 + 兜底。
3. **登记顺序**：加进 `AmbiguityConstants.STAGE2_RULES`。
4. **路由**：在 `Stage2Node.computeRulesToCheck` 的**理解类集合**（现为 `{B000001, B000003}`）里加上它——即"消歧失败时"才进。
5.（可选）若与已有理解类冗余，在 §9 的"理解类内部只报一条"里排好优先级。

> 这里 stage1 与理解类规则的耦合是**有意设计**：stage1 本质就是"理解类规则的统一前置闸"。理解类增减自然要同步那张封闭清单——这正是两阶段口径不打架、结果稳定的保证。

### 12.4 改完后的自检清单

- [ ] 新规则的片段里有没有"拿不准默认不报/默认成功"兜底？（守住单调性，只减不增误报）
- [ ] `STAGE2_RULES` 是否登记？顺序是否符合预期？
- [ ] `Stage2Node` 路由是否把它归到了正确的类（理解类/规范类）？
- [ ] 若是理解类：stage1 是否加了对应 Fn？两边是否一一对齐？
- [ ] §9 组合治理是否需要更新（优先级/抑制）？
- [ ] 用真实数据灰度，确认它只"按预期多报"、没污染其他规则。

## 13. 待优化（Backlog）

以下为已识别、暂未实现的优化项，按需再做：

1. **规则片段外部化 / 可配置**（优先）：当前 `prompts/rules/B0000xx.txt` 打包在 jar 内，改规则需重新打包。可改为：
   - 从**外部目录**加载（如 `ambiguity.rules.dir=/etc/datacheck/rules`），支持热更新、不重新打包即可调规则；
   - 增加**规则启用开关**（如 `ambiguity.rules.enabled=[B000002,B000004]`），由配置决定哪些规则参与 stage2 拼接，便于灰度/按场景启停；
   - 缺失片段时回退到 classpath 内置版本。
2. **门控词表外部化**：高风险词表、备用词表、业务限定四类示例目前硬编码在 `AmbiguityGate` / 规则片段里，可抽到配置文件，便于业务侧自助维护。
3. **缓存主键压缩**：`cache_key CHAR(32)` → `BINARY(16)`（原始 md5 字节），索引减半、IN 查询更快；应用侧把 hex 改为 16 字节。
4. **批内大体量优化**：单次上传极大时，`WHERE cache_key IN(...)` 可替换为临时表 JOIN 批量查（主键 IN 分批通常已够，按需再上）。
5. **结果导出/审核闭环**：缓存表可加"人工修正"列，人工复核后回写，下次直接采用人工结论。
