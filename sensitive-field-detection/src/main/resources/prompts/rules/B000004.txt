## B000004 - 字段中文名英文缩写不明确

**前提**：本规则由代码门控（hitB4）路由，field_cn 已确认含英文字母数量 >= 2。

**适用范围**：
仅适用于 field_cn 本身含有英文缩写的情况。table_en 或 field_en 中的缩写不触发。

**判定标准**：
field_cn 中的缩写，即使结合所有信息（table_cn / table_en / field_en / 缩写映射），仍有多种合理展开，且不同展开指向不同业务含义。

**排除条件（按顺序执行，命中即停止，不再继续判断）**：

1. **语义单元收窄**：先看缩写所在的完整语义单元（缩写 + 周围中文词），
   而不是孤立地看缩写本身。若周围中文已将缩写的含义收窄到唯一，
   映射中其他展开在此语义单元下不构成合理解释 → 不报。
   缩写映射仅在缩写孤立、周围中文无法收窄含义时才有裁决权。

2. **常用缩写**：在当前语境（IT/金融/保险）下，该缩写只有一种被业界普遍接受的理解，
   且这种理解不需要"展开"就能直接使用 → 不报。
   此条优先于映射的多展开结果：即使映射列出了其他展开，
   若这些展开在当前语境下明显不合理，仍视为常用缩写，不报。

3. **缩写映射白名单**：映射已确认唯一含义 → 不报。

4. **中英文对照标注**：field_cn 中缩写与中文以"中文-英文"或"中文(英文)"格式配对出现，
   中文已锁定缩写含义 → 不报。

5. **枚举取值/渠道标签场景**：缩写并非修饰字段核心含义的成分，而是充当某个
   **枚举取值项**或 `标签:内容` / `标签=内容` 结构里的**渠道/取值标签**
   （如 `ivr:手机号`、`webchat:客户姓名` 中的 ivr / webchat 是渠道标识）→ 不报。
   判断要点：去掉该缩写，字段"大致在存什么"不变 → 它只是取值/渠道标签，不背 B000004。
   注：此类"多渠道多值拼接"是命名规范问题，交由命名规范类规则处理，不在 B000004 判。

**示例**：
- 报：field_cn=`TP金额` → Third Party? Transaction Price? 多种展开指向不同业务
- 不报：field_cn=`WP客户端类型`（WP=豁免，白名单已确认唯一含义）
- 不报：field_cn=`产品代码`, table_en=`LLDAC` → 缩写在 table_en 中，不在 field_cn 中
- 不报：field_cn=`账号类型（理赔-claim/保全-pos）` → 中文与英文已配对标注，含义明确
- 不报：field_cn=`OTP验证码` → OTP=One Time Password，业内公认含义
- 不报：field_cn=`原始客户姓名 ivr:手机号 微信:昵称 webchat:客户姓名` → ivr / webchat 是渠道取值标签，非字段核心语义缩写（多值拼接属命名规范问题）

**拿不准时默认不报。**
