。/# 安全分类分级 — 召回与判定优化重构方案

> 关联文档：`安全分类分级接口.txt`（现状设计）。本文件只描述"改什么、为什么、怎么改"，不替代原设计文档。
> 适用规模：关键项目录约 **300 个四级叶子**（chunk）。规模小是本方案很多取舍的前提。

---

## 1. 背景与问题诊断

### 1.1 现状链路

```
输入(单字段) → exemptionCheck(缓存免检) → synonymReplace
             → [vectorSearch ∥ bm25Search] → mergeDedup → aiMatch → resultSave
```

- 索引 `safe_all_topic` 字段：`topic / asset / category / asset_embedding / category_embedding`。
- 向量检索：对 `columnChnName` 求 embedding，匹配 `asset_embedding`（只 embed 了**叶子名**）。
- BM25：`columnChnName` 匹配 `asset` 一个字段。
- 判定：候选只带 `category / asset / vectorScore / bm25Score / source` 给 LLM。

### 1.2 四个根因

| # | 问题 | 现象 | 根因 |
|---|------|------|------|
| P1 | **最强信号未入库** | "客户姓名"召不回枚举了"姓名"的那条目录 | 目录 `content`（枚举示例）、各级定义**根本没进索引**，向量只 embed 了叶子名 |
| P2 | **隐含信息丢失** | 表名/英文名里的语义没用上 | 生产 query 只用 `columnChnName`（`recall_eval.py` 已验证 multifield/rewrite 更好，但没上生产） |
| P3 | **近义兄弟误配** | 客户姓名 ↔ 投保人姓名 ↔ 员工姓名 分不清 | "主体"信息不在字段名里；判定层又看不到定义/content，只能靠分数猜 |
| P4 | **判定层无证据** | LLM 闭卷选择 | 候选未携带路径/定义/content |

### 1.3 结论

不推翻"混合召回 + LLM 判定"的大框架。**300 条规模下不需要 top-down 分层下钻**（会引入误差积累且无必要）。核心是补齐信号 + 提前锁定主体。

---

## 2. 目标与总体思路

四件正交的改造，各治一个根因：

1. **目录索引重构**：把 `content` + 各级定义接入索引（治 P1）。
2. **表级含义推断**：整表一次推断，锁定"主体+规范含义"，注入召回词（治 P2、P3）。
3. **慷慨召回**：用推断词做混合检索，扩大 topK（300 条召回无压力）。
4. **满上下文判定 + 难例 loop**：候选带完整路径/定义/content，主体一致性约束；仅难例触发受控迭代（治 P3、P4）。

**不做**：top-down 贪心分层、beam search（300 条无需剪枝，且会引入级联误差）。层级只作为**判定时的消歧上下文**，不作为硬门。

### 2.1 目标链路

```
[按表聚合输入]
   → tableInfer(整表1次LLM：推断表主体 + 每字段规范含义/主体)   ← 新增
   → 逐字段:
        exemptionCheck(缓存免检)
        → synonymReplace(改用推断后的规范检索词)                ← 改造
        → [vectorSearch ∥ bm25Search](慷慨 topK)               ← 参数调整
        → mergeDedup(候选补齐 content/定义/路径)                 ← 改造
        → aiMatch(满上下文 + 主体一致性)                        ← prompt 改造
        → [hardCaseLoop? 难例才进：重写→重检索→澄清]            ← 新增(可选)
        → resultSave
```

## 3. 详细设计

### 3.1 目录索引重构（治 P1）

> 落地脚本：`eval/build_index.py`（已实现，见 3.1.3）。

#### 3.1.1 源数据结构（真实目录）

目录 Excel 为四级树，层级列为**合并单元格**（一~三级及其定义只在块首行有值，需向下填充）：

| 源列 | 映射到 | 示例 |
|------|--------|------|
| 一级子类 | `level1` / `topic` | 客户 |
| 二级子类 | `level2` | 个人 |
| 二级子类定义说明 | `level2_definition` | 指金融业机构提供各种业务服务的自然人对象相关的各类信息。 |
| 三级子类 | `level3` | 个人自然信息 |
| 三级子类定义说明 | `level3_definition` | 指个人的自然属性信息…… |
| 四级子类 | `asset`（叶子） | 个人基本概况信息 |
| 内容 | `content`（**最强信号，枚举示例**） | 指个人基本情况数据，如个人姓名、性别、证件号码…… |
| 最低安全级别参考 | `security_level` | 3 |

完整 `category` = `level1|level2|level3|level4`，例：`客户|个人|个人自然信息|个人基本概况信息`。

#### 3.1.2 索引 Mapping（新建 `safe_all_topic_v2`）

| 字段 | 类型 | 说明 |
|------|------|------|
| `doc_type` | keyword | `category` / `example`，区分两类文档 |
| `topic` | keyword | 一级（=level1） |
| `level1/level2/level3` | keyword | 各级名（用于聚合/过滤/主体判断） |
| `asset` | text + `asset.kw`(keyword) | 四级叶子名 |
| `category` | keyword | 完整路径，融合去重与最终输出用 |
| `content` | text(中文分词) | 枚举示例原文 |
| `example` | text(中文分词) | 单个枚举示例（仅 example 文档） |
| `level2_definition` / `level3_definition` | text | 各级定义，判定时作消歧上下文 |
| `security_level` | integer | 随目录带出，命中后可直接回填输出 |
| `asset_embedding` | dense_vector(1024, cosine) | 向量字段，两类文档共用同一字段名 |

> 分词器：中文文本用 `ik_max_word`（需装 IK 插件）；未装则脚本支持 `ES_ANALYZER=standard` 回退。

#### 3.1.3 两类文档（治 P1 的核心）

1. **category 文档**（每条四级叶子 1 个）：
   - 向量文本 `_embed_text = asset + "。" + level3_definition + "。" + content`（信号最全）。
   - `_id = cat::<category>`。
2. **example 文档**（把 `content` 里"如 A、B、C 等"拆开，每项 1 个）：
   - 向量文本 = 示例项本身（如"个人姓名"），使"客户姓名/姓名"能直接命中。
   - 带 `category` 指回其叶子；`_id = ex::<category>::<i>`。
   - 300 条目录拆完约 1~2k 文档，ES 无压力。

两类文档在召回后**按 `category` 归并**（复用 `mergeDedup` 的 category 去重），example 命中等价于其父 category 命中。

- **BM25 检索字段**：`multi_match` 覆盖 `asset + content + example`，专有名词/枚举项可被关键词救回。
- 目录静态，embedding 一次性预生成，后续零成本；灰度用 `_v2`，验证后再切 `elasticsearch.index`，可回滚。

### 3.2 表级含义推断（治 P2、P3）—— 新增 `TableInferNode`

**前置改动：输入按表聚合。** 现在 `SecurityClassificationInputDTO` 是逐字段的；表级推断需要整表字段清单。

- 新增批量入口：按 `systemName + tableName` 分组，先对每张表跑一次推断，再展开逐字段流程。
- 若上游无法一次给整表字段，则该节点**降级为逐字段**（只用表名+字段名做轻量补全），不阻断流程。

**输入**：`systemName / systemDesc / tableName / tableChnName` + 该表**所有字段**的 `columnName / columnChnName`（有采样数据更佳，见 3.2.3）。

**输出**（每字段）：
```json
{
  "tableSubject": "投保人信息",
  "fields": [
    {
      "columnName": "xm",
      "normalizedName": "投保人姓名",   // 规范化检索词
      "subject": "投保人",              // 主体（用于判定一致性约束）
      "attribute": "姓名",
      "confidence": "high"             // high/low；low 时召回回退原始词
    }
  ]
}
```

**接入**：
- `normalizedName` 写入 `queryTerms`，供召回使用（替代原始 `columnChnName`）。
- `subject` 存入 `result`，供判定层做主体一致性约束。
- `confidence=low` 的字段：召回时**原始词 + 推断词都用**（expand），避免推断错带偏。

#### 3.2.1 Prompt 要点（`prompts/table-infer.txt`）
- 只基于"表名 + 字段清单（+采样）"推断，**禁止凭空发挥**；拿不准就标 `confidence=low` 并保留原义。
- 展开英文缩写、补全主体、剥离无意义单位/编号。
- 严格 JSON 输出。

#### 3.2.2 成本
一张表几十字段仅 **1 次 LLM 调用**，摊到每字段可忽略；批量场景友好。

#### 3.2.3 采样数据（可选增强）
若合规允许，附几行真实数据（"姓名"列的值像人名、"日期"列像日期），推断准确率更高。涉及隐私/脱敏，默认不启用，配置开关控制。

### 3.3 召回改造

- **query 来源**：改用 `TableInferNode` 的 `normalizedName`；`confidence=low` 回退为"原始词 + 推断词"（`synonym.mode=expand` 语义）。
- **topK**：300 条规模，`vector-search.top-k` 与 `bm25-search.top-k` 提到 **20~30**，`num-candidates` 提到 200~300（几乎全量候选池，`recall@30` 极高）。
- **min-score**：评测期先设 0 看全召回，再回调（现值 0.65 偏高，可能误杀）。
- **融合**：沿用按 `category` 去重；建议引入 RRF（`recall_eval.py` 已有实现）替代当前"取 max score"，排序更稳。

### 3.4 判定改造（治 P3、P4）—— `aiMatch` prompt 升级

**候选携带信息扩充**（`CandidateItem` 增字段）：
```
category(完整路径) / asset / content / level3_definition / vectorScore / bm25Score / source
```

**Prompt 新增规则**：
1. 结合每个候选的 `content` 枚举示例与三级定义判断，而非只看分数。
2. **主体一致性**：字段推断主体（如"投保人"）与候选路径主体不符的，**不得选中**（除非无其他候选且语义高度吻合）。
3. 层级作为消歧上下文：同名叶子（"姓名"）优先选路径主体与字段主体一致者。
4. 都不符合 → `category=null`。

**输出**：沿用 `SecurityMatchResponse`（`category / matched_catalog / reason`）；可加 `security_level / regulatory_level`（从命中目录直接带出，无需 LLM 生成）。

### 3.5 难例 Agent loop（可选，只兜难例）—— `HardCaseLoopNode`

**默认关闭**，通过 `security-classification.hard-case-loop.enabled` 开启。

**触发条件（任一）**：
- `aiMatch` 返回 `category=null`；
- top1 与 top2 的融合分差 < 阈值（近义兄弟）；
- 判定主体与推断主体冲突；
- 候选数为 0。

**动作空间（受限、可枚举）**：
1. 重写 query（换主体表述 / 展开缩写 / 借表上下文补全）；
2. 重新检索（放宽 min-score / 增大 topK / 改用 example doc 检索）；
3. 用候选 `content` 定义做澄清式二次判定。

**终止条件**：命中高置信 / 达最大轮数（**2~3**）/ 明确判 null。

> 定位：主路径覆盖绝大多数字段；loop 只在刀刃上花额外 token，保证成本可控、可回归。

## 4. 工作流编排改动（`SecurityClassificationGraphConfig`）

新增节点与边（保留并行召回结构）：

```
START → tableInfer → exemptionCheck → synonymReplace
      → (vectorSearch ∥ bm25Search) → mergeDedup → aiMatch
      → [条件边] hardCaseLoop --(未解决且轮数未满)--> synonymReplace   // 回边
                              --(已解决/达上限)--> resultSave
      → resultSave → END
```

- `tableInfer` 放在表级（批量入口调用一次），或作为字段级节点读取表级缓存结果。
- `hardCaseLoop` 用条件边 + 回边实现受限迭代；未开启时直接 `aiMatch → resultSave`。
- 新增 state 键：`tableSubject`、`fieldSubject`、`loopRound`，均 `ReplaceStrategy`。

## 7. 兼容与降级

- `TableInferNode` 失败/无整表输入 → 降级逐字段轻量补全，用原始 `columnChnName` 召回，流程不断。
- Embedding/ES 失败 → 沿用现有降级（向量空、仅 BM25）。
- `hard-case-loop.enabled=false` → 行为等同现状 + 前面三项增强，风险最小。
- 建议新建 `safe_all_topic_v2` 灰度，验证通过再切换 `index`，可随时回滚。

## 8. 评测方案（复用 `eval/recall_eval.py`）

- **召回侧**：用标注集跑 `recall@25`，对比 `baseline / multifield / rewrite / +tableInfer / +content入库` 各策略增量。重点看 P1（content 入库）和 P3（表级推断）带来的 recall 提升。
- **判定侧**：`judge_acc(召回命中子集)`、`end2end_acc`。重点看主体一致性规则对"客户/投保人/员工 姓名"这类样本的纠错率。
- **难例 loop**：单独统计触发率、loop 后纠正率、额外 token 成本。
- 建议构造一个"近义兄弟"专项标注小集（各种"姓名"），针对性回归 P3。

## 9. 分阶段落地（按性价比排序）

| 阶段 | 内容 | 治 | 预期收益 | 成本 |
|------|------|----|---------|------|
| S1 | `content`/定义入库 + BM25 multi_match + topK 调大 | P1 | 召回大幅提升 | 低（改索引+检索） |
| S2 | 判定 prompt 带 content/路径 + 主体一致性 | P4/P3 | 误配下降 | 低（改 prompt+候选字段） |
| S3 | 表级含义推断 `TableInferNode` + 输入按表聚合 | P2/P3 | 主体消歧、劣质名归一 | 中（新节点+输入结构） |
| S4 | 难例 `HardCaseLoopNode`（默认关，灰度开） | 兜底 | 难例补救 | 中（新节点+回边） |

**建议先做 S1+S2**（改动小、见效快、覆盖 P1/P4），用评测验证后再上 S3，S4 视残留难例比例决定是否启用。
